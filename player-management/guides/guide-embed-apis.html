Embed API를 사용하는 이유는 무엇입니까?

Embed API를 사용하면 특정 플레이어의 여러 인스턴스를 만들 수 있습니다. 이 플레이어/인스턴스 관계를 부모/자식 관계로 생각하는 좋은 방법입니다. 단일 플레이어는 상위 플레이어이고 Embed API로 생성된 플레이어는 상위 플레이어의 하위 플레이어입니다. 상위 플레이어에는 플레이어가 갖고자 하는 대부분의 속성이 있으며, Embed API를 사용하여 다른 하위 플레이어의 속성 하위 집합을 사용자 지정할 수 있습니다. 예를 들어, 다른 미디어를 로드하거나 다른 하위 플레이어와 함께 다른 플러그인 및 스타일을 사용할 수 있습니다.

다음 다이어그램은 기능을 명확히 하는 데 도움이 됩니다. 부모 아래 왼쪽에 표시되고 두 명의 하위 플레이어가 오른쪽에 표시됩니다. 다음 사항을 준수하십시오.

  • 포스터는 두 자녀에게 상속됩니다.
  • 재생 버튼의 모양은 상위 하위에 상속되고 하위 하위에서 재정의됩니다.
  • 상위 자식은 속성을 추가합니다. 이 경우에는 부모에게 없는 오버레이가 있습니다.

이 부모/자식 관계의 또 다른 강력한 기능은 상속이 진행 중이라는 것입니다. 다음 다이어그램은 상위에 할당된 새 포스터를 보여주며 두 하위 모두 해당 구성 변경을 상속합니다.

메모: 기본적으로 각 상위 플레이어에 대해 50명의 하위 플레이어로 제한됩니다. 더 많은 사용 사례가 있는 경우 지원팀에 문의하여 상황을 논의하세요.

Embed API를 사용하지 않는 경우

사용 사례에 필요한 경우 Embed API를 사용해야 하는 몇 가지 좋은 이유가 있지만 일반 플레이어를 고수해야 하는 몇 가지 좋은 이유도 있습니다. 다음은 몇 가지입니다.

  • 하위 플레이어는 Video Cloud Studio를 사용하여 편집할 수 없습니다. 플레이어 관리 API를 통해서만 하위 플레이어를 편집할 수 있습니다. Video Cloud Studio에서 하위 플레이어의 상위 플레이어를 편집할 수 있지만 상위 플레이어에 대한 변경 사항은 모든 하위 플레이어에 영향을 미칩니다.
  • 상위 플레이어와 연결된 하위 플레이어가 많은 경우 상위 플레이어를 게시하는 데 시간이 오래 걸릴 수 있습니다. 각 하위 플레이어는 별도로 게시되며 30명 이상의 하위 플레이어가 있는 경우 하위 플레이어 게시가 약간 지연될 수 있습니다. 이것은 30명의 일반 플레이어를 동시에 게시하는 것과 정확히 동일한 경우입니다.

위의 이유를 감안할 때 일반 플레이어를 사용하여 시작한 다음 하위 플레이어가 필요하다고 판단되면 삽입을 시도하는 것이 합리적일 수 있습니다.

비디오 태그 데이터 포함

부모와 자식 플레이어의 표기법 차이가 있습니다. 표준 인페이지 삽입 플레이어 코드는 다음 형식으로 표시됩니다.

    <video-js
      data-account="1507807800001"
      data-player="HiAdwRZ7kK"
      data-embed="default"
      controls=""
      data-application-id=""
      class="vjs-fluid"></video-js>

NS data-embed속성은 플레이어가 부모인지 자식인지를 결정합니다. 값이default , 플레이어는 부모입니다. 플레이어가 어린이일 경우, data-embed속성에는 상위 플레이어의 ID가 포함됩니다. 그 예는 다음과 같습니다.

NS data-embed속성은 플레이어가 부모인지 자식인지를 결정합니다. 값이default , 플레이어는 부모입니다. 플레이어가 어린이일 경우, data-embed속성에는 상위 플레이어의 ID가 포함됩니다. 그 예는 다음과 같습니다.

하위 플레이어 URL

부모 플레이어와 자식 플레이어를 어떻게 구별합니까? URL이 다릅니다. 예를 들어 상위 플레이어의 URL은 다음과 같습니다.

    //players.brightcove.net/1507807800001/HiAdwRZ7kK_default/index.min.js

Embed API를 사용하여 하위 플레이어를 만든 후 다음과 같이 하위 플레이어의 ID가 상위 URL에 추가됩니다.

    //players.brightcove.net/1507807800001/HiAdwRZ7kK_NURK56ZSV/index.min.js

부모/자식 사용 사례

여러 비디오 플레이어를 사용한다고 가정합니다. 종종 플레이어의 공통 기능은 거의 동일하지만 몇 가지 경우에는 특별한 경우에 대해 플레이어를 조정하려고 합니다. 다음과 같이 플레이어 구성 API를 사용하여 여러 플레이어를 만들 수 있습니다. POST그리고PATCH그러나 이것은 심각한 유지 관리 문제로 이어질 수 있습니다. 예를 들어 모든 플레이어의 포스터를 변경하고 싶다고 가정해 보겠습니다. 이것은 사용하는 것을 의미합니다PATCH모든 다른 플레이어에. 반면에 하위 플레이어를 만들면PATCH상위 플레이어와 모든 하위 플레이어는 자동으로 새 포스터를 갖게 됩니다.

생성 과정

했다면단계별: 선수 관리플레이어 관리 API에 HTTP 메서드를 전달하기 위해 curl 문을 사용하는 프로세스를 보았습니다. 여기에서도 동일한 접근 방식이 사용됩니다.

플레이어를 생성하기 위해 다음과 같은 플레이어 구성 API와 함께 몇 가지 HTTP 메서드를 사용했을 가능성이 큽니다.

  • 다음을 사용하여 플레이어 생성POST에게 https://players.api.brightcove.com/v2/accounts/ {account_id} /플레이어
  • 다음을 사용하여 플레이어를 업데이트하십시오. PATCH에게 https://players.api.brightcove.com/v2/accounts/ {account_id} /플레이어/ {player_id} /구성
  • 다음을 사용하여 업데이트된 플레이어를 게시합니다. POST에게 https://players.api.brightcove.com/v2/accounts/ {account_id} /플레이어/ {player_id} /게시

Embed API를 사용하는 하위 플레이어에도 유사한 접근 방식이 사용됩니다. 매우 높은 수준에서 다음을 수행합니다.

  • 다음을 사용하여 자식 플레이어를 만듭니다. POST에게 https://players.api.brightcove.com/v2/accounts/ {account_id} /플레이어/ {player_id} / 임베딩. 참고: Embed API로 생성된 하위 플레이어는 생성 시 자체 게시하므로 하위 플레이어 생성 시 게시할 필요가 없으며 하위 플레이어 업데이트 시에만 게시합니다.
  • 다음을 사용하여 하위 플레이어를 업데이트합니다. PATCH에게 https://players.api.brightcove.com/v2/accounts/ {account_id} /플레이어/ {player_id} / 삽입/$EMBED_ID/구성
  • 다음을 사용하여 하위 플레이어를 게시합니다. POST에게 https://players.api.brightcove.com/v2/accounts/ {account_id} /플레이어/ {player_id} / 삽입/$EMBED_ID/게시

다음 내용은 프로세스를 자세히 설명합니다.

자식 플레이어 만들기

자식 플레이어를 만들려면 HTTP를 사용합니다. POST방법:

    curl /
    --header "Content-Type: application/json" /
    --user $EMAIL /
    --request POST /
    --data '{
    "media": {
    "sources": [
      {
        "src":"http://solutions.brightcove.com/bcls/assets/videos/BirdsOfAFeather.mp4",
        "type":"video/mp4"
      }
    ],
    "poster": {
      "highres":"http://solutions.brightcove.com/bcls/assets/images/BirdsOfAFeather.jpg"
    }
    }
      }' /
    https://players.api.brightcove.com/v2/accounts/{account_id}/players/$PLAYER_ID/embeds

하위 플레이어 생성에 대한 예시 응답은 다음과 같습니다.

이제 사용할 수 있습니다. url결과를 보려면 자식 플레이어에서 속성을 가져옵니다. 아래 예에서 하위 플레이어는 단계별로 생성된 상위 플레이어에 추가되었습니다. 플레이어 관리. 새 포스터와 비디오가 표시되지만 상위 플레이어의 오버레이 플러그인은 여전히 존재합니다.

하위 플레이어 업데이트

자식 플레이어를 업데이트하려면 HTTP를 사용합니다. PATCH방법. 다음 curl 문은 다음을 업데이트합니다. poster재산. 설정했다고 가정합니다. $EMBED_ID적절하게 환경 변수:

    curl
    --header "Content-Type: application/json"
    --user $EMAIL
    --request PATCH
    --data '{
    "media": {
    "poster": {
      "highres":"http://solutions.brightcove.com/bcls/assets/images/Water-Splashing.jpg"
    }
    }
      }'
    https://players.api.brightcove.com/v2/accounts/{account_id}/players/$PLAYER_ID/embeds/$EMBED_ID/configuration

응답은 두 가지 모두에 대한 미리보기 정보를 제공합니다. preview_url그리고preview_embed_code암호:

    {
        "preview_url": "http://preview-players.brightcove.net/1507807800001/668c5107-a80c-4940-8c17-279c01ce101c/be864624-8d85-4dfc-8fe6-4e9dd4c70417/index.html",
        "preview_embed_code": "<iframe src='//preview-players.brightcove.net/1507807800001/668c5107-a80c-4940-8c17-279c01ce101c/be864624-8d85-4dfc-8fe6-4e9dd4c70417/index.html' allowfullscreen webkitallowfullscreen mozallowfullscreen></iframe>"
    }

하위 플레이어 게시

자식 플레이어가 변경되면 게시해야 합니다. 확인하십시오$EMBED_ID환경 변수가 설정되면 새로 변경된 하위 플레이어를 게시할 수 있습니다.

    curl
    --header "Content-Type: application/json"
    --user $EMAIL
    --request POST
    https://players.api.brightcove.com/v2/accounts/{account_id}/players/$PLAYER_ID/embeds/$EMBED_ID/publish

응답은 플레이어를 게시하는 것과 매우 유사하게 하위 플레이어를 사용하는 데 필요한 중요한 정보를 제공합니다.

        "embed_code": "<iframe src='//players.brightcove.net/1507807800001/668c5107-a80c-4940-8c17-279c01ce101c_be864624-8d85-4dfc-8fe6-4e9dd4c70417/index.html' allowfullscreen webkitallowfullscreen mozallowfullscreen></iframe>",
        "embed_in_page": "http://players.brightcove.net/1507807800001/668c5107-a80c-4940-8c17-279c01ce101c_be864624-8d85-4dfc-8fe6-4e9dd4c70417/in_page.embed"

자녀 정보 표시

HTTP를 사용할 수 있습니다. GET자식 플레이어에 대한 정보를 검색하는 메서드입니다. curl 문 예시는 다음과 같습니다.

    curl
      --header "Content-Type: application/json"
      --user $EMAIL
      --request GET
      https://players.api.brightcove.com/v2/accounts/{account_id}/players/$PLAYER_ID/embeds

상당히 많은 양의 JSON 데이터가 반환됩니다.

하위 플레이어 삭제

다음을 사용하여 하위 플레이어를 삭제할 수도 있습니다. DELETE방법. 다음은 자식 플레이어를 삭제하는 curl 구문의 예입니다.

    curl
    --header "Content-Type: application/json"
    --user $EMAIL
    --request DELETE
    https://players.api.brightcove.com/v2/accounts/{account_id}/players/$PLAYER_ID/embeds/$EMBED_ID

물론 이것은 상위 플레이어가 아닌 하위 플레이어에만 영향을 미칩니다.
