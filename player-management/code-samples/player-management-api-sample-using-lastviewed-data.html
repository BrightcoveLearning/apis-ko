이 샘플 앱은 Player Management API의 기본 사항을 탐색하기 위한 방법입니다. 엔터프라이즈용 배포 가능한 앱이 아닙니다. 예를 들어 양식 유효성 검사는 완전하지 않으며 암호는 HTTPS를 사용하지 않고 전송됩니다.

last_viewed 정보 이해하기

각 플레이어는 플레이어가 로드된 횟수에 대한 레코드가 포함된 배열과 연결되어 있습니다. 이 정보는 배열 이름에 저장됩니다. last_viewed . 아래에서 그러한 배열의 예를 볼 수 있습니다. 참고로count특정 플레이어가 하루에 로드된 시간을 기록합니다. NS date특정 날짜를 지정합니다.

    [
      {
        "count": 13,
        "date": "2018-10-24T05:00:00.000Z"
      }, {
        "count": 22,
        "date": "2018-10-23T05:00:00.000Z"
      }, {
        "count": 40,
        "date": "2018-10-22T05:00:00.000Z"
      }, {
        "count": 2,
        "date": "2018-10-21T05:00:00.000Z"
      }
    ]
다음 사항에 유의하십시오.
  • 카운트는 하루에 한 번 컴파일됩니다.
  • 카운트는 플레이어 로드를 위한 것이며 카운트에 추가하기 위해 비디오가 실제로 재생될 필요는 없습니다.
  • 배열은 30개 항목으로 제한됩니다. 이는 연속 30일이 아니라 카운트가 0이 아닌 30일일 가능성이 높습니다.
  • 컴파일된 데이터는 2018년 10월 중순 이전의 개수를 기록하지 않습니다.
  • 플레이어에 데이터가 없으면 배열이 비어 있습니다.

앱 아키텍처

이 앱의 상위 수준 응용 프로그램 논리는 다음과 같습니다.

  1. 계정의 모든 플레이어에 대한 정보를 검색하는 데 필요한 사용자 데이터 수집
  2. 플레이어 생성을 위해 API에 전달할 JSON 데이터 구조 빌드
  3. 버튼 클릭 시 AJAX 호출을 수행하여 계정의 모든 플레이어에 대한 정보 검색
  4. AJAX 호출 성공 시 추출마지막으로 본각 플레이어에 대한 배열입니다.
  5. 각 플레이어에 대해 표시마지막으로 본테이블의 정보
