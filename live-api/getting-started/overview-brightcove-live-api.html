  • 서버 측 광고 삽입( 싸이 )
  • AES-128암호화
  • 라이브 스트림에서 가져온 클립에서 주문형 비디오 자산 만들기
  • DVR능력
  • 다수의 CDN

채널 및 이벤트 시간

Live에는 두 가지 구매 옵션이 있습니다.

  • 스트리밍 시간의 구매 이벤트 시간
  • 스트리밍 채널 구매

이벤트 스트리밍 시간과 채널을 모두 구매할 수도 있습니다. 오퍼링에 대한 자세한 내용은 계정 관리자에게 문의하십시오.

이벤트 스트리밍 시간과 채널을 모두 구매한 경우 , 다음 사항에 유의해야 합니다.
  • 기본적으로 라이브 작업은 이벤트 스트리밍 시간을 기준으로 청구됩니다.
  • 대신 채널 작업으로 비용이 청구되도록 하려면 다음을 수행하십시오. ~해야하다라이브 작업을 생성할 때 요청 본문에 다음 필드를 포함합니다.

    "channel_type": "24x7"

  • 채널 전용 고객은 라이브 작업을 생성할 때 항상 요청 본문에 다음 필드를 포함해야 합니다.

    "channel_type": "24x7"

  • 스트리밍 시간의 이벤트 시간만 구매한 고객은 반드시 ~ 아니다포함하다"channel_type": "24x7"요청에.

토큰 인증

Brightcove는 라이브 비디오 스트림 재생 URL에 토큰 인증을 추가하는 옵션을 제공합니다. 토큰 인증을 추가하려면, Brightcove 지원에 문의 . 토큰 인증을 설정하는 데 최대 3일이 소요될 수 있습니다.

토큰의 TTL(time-to-live)은 1시간에서 365일 사이의 값으로 설정할 수 있습니다. TTL을 설정하는 기간은 배포하는 라이브 스트림의 종류에 따라 다릅니다. 그러나 TTL은 계정 전체 설정이며 모든 실시간 스트림에 적용됩니다.

입력 프로토콜

Brightcove Live는 여러 입력 프로토콜을 지원합니다. 사용protocol사용할 작업을 지정하기 위해 작업을 생성할 때 요청 본문의 필드입니다. 지원되는 값은 다음과 같습니다.

  • rtmp (기본값)
  • rtp
  • rtp-fec
  • srt

RTMP 프로토콜은 FLV 형식의 스트림을 전달하기 위한 것입니다. 다른 프로토콜은 MPEG2-TS를 전달하기 위한 것입니다.

사용하는 경우rtp , rtp-fec또는srt , 다음도 지정해야 합니다. cidr_whitelist (보다클래스 없는 도메인 간 라우팅 ).

사용하는 경우rtmp , 다음을 지정할 수 있습니다. ip_whitelist대신 입력용으로 사용되지만 필수는 아닙니다.

RTP+FEC 작업에 대한 요청 본문 예:

{
    "live_stream":true,
    "region":"us-west-2",
    "reconnect_time":300,
    "outputs":[
      {
        "label": "hls720p",
        "live_stream": true,
        "height": 720,
        "video_bitrate": 800,
        "segment_seconds": 6,
        "keyframe_interval": 90
      }
    ],
    "protocol": "rtp-fec",
    "cidr_whitelist": ["127.0.0.1/32"]
}

정적 진입점

정적 진입점(SEP) 기능을 사용하면 진입점 URL과 재생 URL을 정적 및 재사용 가능한 상태로 유지하면서 활성화 및 비활성화할 수 있는 장기 실행 라이브 작업을 허용합니다. 이 기능을 통해 고객은 자신의 시설이나 현장에서 인코더를 구성할 수 있고 고객이 라이브 채널 또는 프로그램에 대한 자체 스케줄링 로직을 생성할 수 있습니다. 보다정적 진입점자세한 내용은.

캡션

캡션이 h264 입력 신호(user_data 패킷에서 올바르게 신호됨) 내부에 있으면 h264 출력으로 전달됩니다.

방송 Elemental 라이브 인코더를 사용하는 경우 SDI(EIA-608/CEA-608) 또는 기타 소스(SCTE-20, SCC, Teletext, DVB-Sub, Ancillary, ARIB, TTML, SCTE-27, STL, SRT, SMI)를 저장하고 저희에게 보내는 h264 스트림에 넣습니다. 다른 방송 등급 인코더도 동일한 작업을 수행할 수 있지만 공식적으로 테스트하지는 않았습니다.

현재 제한 사항으로 Safari는 표시할 캡션이 없는 경우에도 라이브 스트림에 대한 CC 컨트롤을 표시합니다.
