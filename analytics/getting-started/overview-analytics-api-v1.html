캐싱

성능상의 이유로 API 응답은 약 5분 동안 캐시되지만 정확한 시간은 여러 요인에 따라 다를 수 있습니다. 어떠한 것도애널리틱스 API쿼리에서 응답 헤더에서 캐시에 대한 정보를 얻을 수 있습니다.

NS Cache-Control결과가 캐시될 최대 시간을 초 단위로 알려줍니다(위의 예에서는 24초). NS Last-Modified그리고Expires헤더는 현재 캐시가 생성된 시점과 만료 시점을 알려줍니다.

대부분의 경우 이는 문제가 되지 않을 수 있지만 분석 데이터의 최신 상태가 매우 중요한 경우 쿼리가 더 오래 실행될수록 더 오래 캐시되며 실시간(비조정 시간별) 데이터만 가져오는 것으로 보고합니다. 조정된 데이터를 가져오는 한 캐시되지 않습니다(실시간 데이터에만 또는 추가로). 을 찾다실시간 및 조정된 데이터에 대한 완전한 설명당신이 좋아한다면; 짧은 버전은애널리틱스 API두 가지 데이터 버킷에 의존합니다.

  • 즉시 사용 가능하고 32일 동안 저장되는 실시간 또는 시간별 조정되지 않은 데이터
  • 영구적으로 저장되는 조정된 데이터 실시간 데이터를 조정하여 정확도를 높이고 24시간마다 조정된 데이터 저장소에 저장합니다.

다음을 사용하여 조정된 데이터 또는 실시간 데이터로 결과를 제한할 수 있습니다. 화해매개변수.

캐싱을 최소화하려면:

  • 사용reconciled=false결과를 실시간 데이터로 제한하는 매개변수
  • 작은 것을 사용날짜 범위 , 전체 범위가 지난 32일 이내에 속하는지 확인하십시오.

시간 초과

Analytics API는 완료되지 않은 경우 8분 후에 시간 초과를 요청합니다. 시간 초과가 8분 미만으로 표시되면 원인은 일부 클라이언트 측 제한입니다.

반품할 수 있는 최대 품목

반품할 수 있는 최대 항목 수는 100만 개입니다. 대부분의 경우 한도에 도달할 가능성은 낮지만 보고서를 요청하는 경우date예를 들어, 장기간에 걸친 차원이 가능합니다. 백만 항목 제한에 도달하면 반환되는 항목 수를 줄이기 위해 요청을 수정해야 합니다. 일반적으로 가장 간단한 방법은 데이터 범위를 줄이는 것입니다( from그리고to매개변수는 나중에 논의됨).

동시 요청

단일 계정은 한 번에 하나의 요청으로 제한됩니다. 여러 동시 요청이 연속적으로 실행됩니다.

예를 들면 다음과 같습니다.

  1. API 요청 "A"를 시작합니다.
  2. 동일한 계정에 대한 API 요청 "B"를 시작합니다.
  3. 요청 "B"는 "A"가 완료될 때까지 완료되지 않습니다.
  4. "A" 요청이 너무 오래 걸리면 "요청이 보류 중입니다. 다시 시도하십시오"라는 오류가 요청 "A"에 수신됩니다.
  5. 요청 "A"가 너무 오래 걸리면 요청 "B"가 동일한 오류를 수신할 수 있습니다. A + B를 완료하는 데 걸리는 시간이 제한 시간 값보다 크면 요청 "B"에 오류가 발생합니다.

동시에 여러 요청을 하는 경우 받은 순서대로 한 번에 하나씩 처리됩니다.

"보류 중인 오류"와 함께 반환되는 요청은 결국 완료되어 캐시에 저장됩니다. 이는 동일한 데이터에 대한 향후 요청이 거의 즉시 반환되지만 5분 캐시가 만료되기 전에 요청이 이루어진 경우에만 반환됨을 의미합니다.

시스템은 2-4분을 기다렸다가 동일한 요청을 다시 수행하여 보류 중인 오류를 처리해야 합니다.

모범 사례

위의 정보를 바탕으로 다음을 모범 사례로 권장합니다.
  1. 에 요청하십시오. 애널리틱스 API앱의 다른 작업이 데이터 없이 진행할 수 있는 경우 비동기식으로
  2. 응답이 다음과 같을 경우 자동으로 요청을 재시도합니다. Processing query in background
  3. 어쨌든 캐시된 데이터를 검색하게 되므로 요청을 5분마다 반복하지 마십시오.
  4. 여러 요청을 동시에 또는 빠르게 연속적으로 수행하는 경우 더 오래 실행되는 요청(가장 많은 항목과 데이터를 반환하는 요청)을 마지막에 만들어 단기 실행 요청이 지연되지 않도록 하십시오.
